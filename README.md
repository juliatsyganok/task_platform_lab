# Платформа автоматизированной обработки задач 

Данный репозиторий содержит комплексный проект распределённой бэкенд-системы, разработанный в рамках учебного курса.  
Платформа спроектирована для гибкого приёма, организации, хранения и параллельного выполнения различных единиц работы *(задач)*.

Архитектура системы построена на принципах **слабой связанности компонентов**, **контрактного программирования** и **высокой расширяемости**, что позволяет добавлять новые источники данных и обработчики логики без изменения существующего ядра кода.

---

## Структура проекта

Репозиторий имеет чёткую модульную структуру, где каждая папка соответствует определённому этапу разработки платформы:

- **`src_lab1/`** — Подсистема приёма данных и интеграция с внешними источниками.
- **`src_lab2/`** — Доменная модель задачи с механизмами безопасной валидации состояний.
- **`src_lab3/`** — Пользовательская коллекция (очередь) для эффективной потоковой обработки.
- **`src_lab4/`** — Асинхронное ядро для конкурентного выполнения задач.
- **`tests/`** — Единый пакет модульных тестов для проверки всех компонентов системы.

## Описание лабораторных работ

###Лабораторная работа №1. Источники задач и контракты (`src_lab1`)

- **Суть работы:**  
  Изучение принципов динамической утиной типизации (Duck Typing) и контрактного программирования.

- **Что сделано:**  
  Описан единый интерфейс источников данных через `typing.Protocol` со специальной проверкой совместимости во время выполнения программы (`@runtime_checkable`).  
  Реализован класс `ApiTaskSource`, который обращается к внешнему веб-сервису через HTTP-запросы (библиотека `requests`), принимает ответы в формате JSON и динамически преобразует их в готовые структуры задач с обработкой сетевых ошибок.

---

### Лабораторная работа №2. Модель задачи: дескрипторы и `@property` (`src_lab2`)

- **Суть работы:**  
  Освоение механизмов управления доступом к атрибутам класса и защита внутренней бизнес-логики от некорректных изменений.

- **Что сделано:**  
  Разработан полноценный класс задачи `Task` с набором атрибутов (идентификатор, описание, приоритет, статус, время создания).  
  Для проверки типов и валидации динамически изменяемых полей спроектированы пользовательские дескрипторы данных.  
  С помощью встроенного декоратора `@property` реализованы вычисляемые свойства, такие как автоматическая оценка готовности задачи к выполнению.

---

### Лабораторная работа №3. Очередь задач: итераторы и генераторы (`src_lab3`)

- **Суть работы:**  
  Проектирование кастомных контейнеров и организация экономичной, ленивой обработки потоков данных.

- **Что сделано:**  
  Создан класс очереди задач `TaskQueue`, полностью поддерживающий протоколы `__iter__` и `__next__` для многократного обхода коллекции стандартными циклами Python.  
  Фильтрация задач по приоритету и статусу реализована через функции-генераторы с ключевым словом `yield`.  
  Это позволяет системе работать с большими объёмами данных в потоковом режиме без выделения избыточной оперативной памяти.

---

### Лабораторная работа №4. Асинхронный исполнитель задач (`src_lab4`)

- **Суть работы:**  
  Построение высокопроизводительных конкурентных приложений на базе неблокирующих вычислений.

- **Что сделано:**  
  Разработано асинхронное исполнительное ядро системы, использующее `asyncio.Queue` для безопасного распределения нагрузки между воркерами.  
  Контракты обработчиков формализованы через протоколы.  
  Жизненный цикл используемых системных ресурсов (сетевые сессии, файлы) строго контролируется асинхронными контекстными менеджерами.  
  Весь код написан с использованием конструкций `async/await`, полностью исключает блокировку главного цикла событий (event loop) и поддерживает централизованное логирование ошибок.

## Тестирование и стандарты качества

Проект разработан с использованием современного менеджера пакетов `uv` (конфигурация описана в `pyproject.toml` и `uv.lock`).  

Все кодовые базы модулей **полностью аннотированы типами** и покрыты модульными тестами, расположенными в директории `tests/`.  

📊 **Общий уровень покрытия кода тестами составляет более 80%**, что гарантирует стабильность работы и архитектурную целостность платформы.
